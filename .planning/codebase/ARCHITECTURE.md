# Architecture

**Analysis Date:** 2026-03-10

## Pattern Overview

**Overall:** Static Site Generation (SSG) with content-driven routing

**Key Characteristics:**
- Astro-based framework for zero-JavaScript-by-default static generation
- Content collection system (Astro Content API) managing all blog posts
- Markdown-first content with frontmatter metadata
- Dynamic route generation from content at build time
- Component-based UI with Astro islands for interactivity
- Tailwind CSS for styling with theme support (light/dark mode)
- Search capability via Pagefind static search index

## Layers

**Content Layer:**
- Purpose: Define and validate content structure
- Location: `src/content.config.ts`, `src/data/blog/`
- Contains: Content collection schemas (Zod), markdown post files
- Depends on: Astro content loader (glob-based file discovery)
- Used by: Page generation pipeline, data utilities

**Page Generation Layer:**
- Purpose: Build pages from content at build time
- Location: `src/pages/`, `src/pages/posts/`, `src/pages/tags/`
- Contains: Astro route components with `getStaticPaths()` for dynamic routes
- Depends on: Content layer (collections), utility functions for filtering/sorting
- Used by: Build process to generate static HTML files

**Layout & Component Layer:**
- Purpose: Reusable UI structure and components
- Location: `src/layouts/`, `src/components/`
- Contains: Layout templates (Layout.astro, PostDetails.astro, Main.astro), UI components (Header, Footer, Card, Pagination, etc.)
- Depends on: Configuration, utilities, styles
- Used by: Page routes and other layouts

**Utility Layer:**
- Purpose: Business logic for content manipulation
- Location: `src/utils/`
- Contains: Post filtering, sorting, tag extraction, path generation, OG image generation
- Depends on: Content types, configuration
- Used by: Page routes, components

**Configuration & Constants:**
- Purpose: Site-wide settings and metadata
- Location: `src/config.ts`, `src/constants.ts`
- Contains: SITE config (title, author, post per page, timezone), SOCIALS and SHARE_LINKS arrays
- Depends on: None
- Used by: All layers

**Style & Script Layer:**
- Purpose: Styling and client-side behavior
- Location: `src/styles/`, `src/scripts/`
- Contains: Global CSS (Tailwind, typography), theme switching script
- Depends on: Tailwind CSS framework
- Used by: Layout.astro (base HTML template)

## Data Flow

**Build-Time Content Processing:**

1. Content Discovery: Glob loader finds all `*.md` files in `src/data/blog/`
2. Validation: Astro validates frontmatter against Zod schema in `src/content.config.ts`
3. Collection: Posts stored in `blog` collection via `getCollection("blog")`
4. Route Generation: Each page route calls `getStaticPaths()` to generate URLs from posts
5. HTML Generation: Astro renders pages combining layouts, components, and content
6. Static Output: Results written to `dist/` directory

**Search Index Generation:**

1. After build completes, Pagefind scans generated HTML in `dist/`
2. Creates searchable index in `dist/pagefind/`
3. Index copied to `public/pagefind/` for deployment

**Data Filtering & Sorting Pipeline:**

```
Raw Posts (all .md files)
  ↓
postFilter (remove drafts, check publish time + margin)
  ↓
getSortedPosts (sort by pubDatetime desc, then modDatetime)
  ↓
Sorted, filtered posts → Components/Pages
```

**Theme State Management:**

1. Inline script in `<head>` reads `localStorage.theme` immediately
2. Sets `data-theme` attribute on `<html>` element before page renders
3. Prevents flash of unstyled content (FOUC)
4. Client-side `theme.ts` script manages theme switching after page load

## Key Abstractions

**Post (CollectionEntry<"blog">):**
- Purpose: Typed representation of a blog post
- Examples: Files in `src/data/blog/`, processed by `src/content.config.ts`
- Pattern: Zod-validated frontmatter + Markdown content body
- Schema includes: author, pubDatetime, modDatetime, title, featured, draft, tags, ogImage, description, canonicalURL, hideEditPost, timezone

**Tag (Tag interface):**
- Purpose: Represent categorized content grouping
- Examples: "discussion", "braindump", "agents" (from post frontmatter tags array)
- Pattern: Normalized tag name + slug form generated by `getUniqueTags()`
- Used for: Tag-based navigation pages and filtering

**Page Routes (Dynamic with getStaticPaths):**
- Purpose: Generate URLs dynamically from content at build time
- Examples: `/posts/[...slug]/index.astro`, `/tags/[tag]/[...page].astro`
- Pattern: Astro's file-based routing + `getStaticPaths()` for dynamic segments
- Returns: Array of { params, props } objects, one per generated page

**Pagination:**
- Purpose: Split large post lists across multiple pages
- Examples: `/posts/`, `/posts/2/`, `/posts/3/` (based on SITE.postPerPage)
- Pattern: Built-in `paginate()` function in `getStaticPaths()`

## Entry Points

**Homepage:**
- Location: `src/pages/index.astro`
- Triggers: Direct URL `/` or site root
- Responsibilities: Render hero section, display featured posts, show recent posts, provide social links and RSS feed link

**Blog Posts List:**
- Location: `src/pages/posts/[...page].astro`
- Triggers: `/posts/`, `/posts/2/`, `/posts/3/` (paginated)
- Responsibilities: Fetch all non-draft posts, paginate by SITE.postPerPage, render post cards with pagination controls

**Post Detail:**
- Location: `src/pages/posts/[...slug]/index.astro`
- Triggers: `/posts/{slug}/` (dynamically generated from post id/filePath)
- Responsibilities: Fetch single post, render with PostDetails layout, provide edit link, share buttons, navigation to prev/next posts

**Tag Page:**
- Location: `src/pages/tags/[tag]/[...page].astro`
- Triggers: `/tags/`, `/tags/{tagName}/`, `/tags/{tagName}/2/` (paginated)
- Responsibilities: Extract unique tags from posts, generate pages per tag, render filtered posts by tag with pagination

**Static Feeds:**
- Location: `src/pages/rss.xml.ts`, `src/pages/og.png.ts`, `src/pages/robots.txt.ts`
- Triggers: `/rss.xml`, `/og.png`, `/robots.txt`
- Responsibilities: Generate feed/metadata files at build time

**Archives:**
- Location: `src/pages/archives/index.astro`
- Triggers: `/archives/` (optional via SITE.showArchives)
- Responsibilities: Display all posts in archive format

## Error Handling

**Strategy:** Build-time validation with graceful degradation

**Patterns:**
- Zod schema validation in `src/content.config.ts` catches invalid post frontmatter at build time
- `postFilter()` excludes draft posts and scheduled posts (not yet published)
- Missing optional fields fall back to defaults (e.g., author defaults to SITE.author)
- Dynamic routes return empty arrays if no content matches criteria
- Client-side: 404.astro handles missing routes

## Cross-Cutting Concerns

**Logging:** No explicit logging layer. Build-time messages via Astro CLI and browser console.

**Validation:** Content validation via Zod schemas in `src/content.config.ts`. Frontmatter structure enforced.

**Authentication:** None (static site). No user login or access control.

**Metadata & SEO:**
- Structured data (JSON-LD) injected in Layout.astro for BlogPosting schema
- Open Graph tags for social sharing (og:title, og:description, og:image)
- Twitter Card metadata
- Canonical URLs via canonicalURL frontmatter field
- Dynamic OG images generated via Satori + Resvg (if SITE.dynamicOgImage enabled)

**Analytics:** Simple Analytics plugin integrated in Layout.astro via `<SimpleAnalytics />` component

**Search:** Pagefind integration for full-text search of generated HTML (no backend required)
